# Day08-Day21 跟读主线

## 先说结论

可以单独看，而且确实应该单独看。

因为 `Day01-Day07` 和 `Day08-Day21` 的关注点不一样：

```python
Day01-Day07 = 手写一个最小 Agent 骨架
Day08-Day21 = 把这个骨架升级成图系统、记忆系统、LLM 系统和工程系统
```

所以你感觉“后面这几天和前 7 天不一样”，这个感觉是对的。

前 7 天更像是在回答：

```python
一个 Agent 最少由什么组成？
```

而 Day08-Day21 更像是在回答：

```python
当 Agent 开始变复杂以后，怎么把它组织成一个更像真实项目的系统？
```

---

## 这一段到底在学什么

你不要把 Day08-Day21 看成“14 天零散内容”。

更好的理解方式是：

```python
它是在给前 7 天的最小 Agent，补上真正的系统能力。
```

这 14 天主要补了 5 类东西：

1. 图和流程编排
2. 记忆和持久化
3. LLM 与工具协作
4. 人工介入、评测、可靠性
5. 配置、日志、毕业项目收口

所以这段主线可以压缩成一句话：

```python
从“能跑一轮”升级到“能作为一个系统持续运行”
```

---

## 第一层：图系统（Day08-Day10）

这一段的核心不是新功能，而是新组织方式。

### Day08

Day08 第一次正式进入 LangGraph。

它解决的问题不是“做什么业务”，而是：

```python
如果一个系统由多个步骤组成，怎么把这些步骤放进图里？
```

所以 Day08 的本质是：

```python
最小图：node + edge + state 流转
```

你可以把 Day08 理解成：

- 以前是你自己手动串函数
- 现在开始让图来管理流程

---

### Day09

Day09 在图里加入“分支”。

它解决的问题是：

```python
系统怎么根据 state 决定下一步走哪条路？
```

比如：

- `intent == weather` 走天气分支
- `intent == math` 走计算分支
- 其他走 unknown 分支

所以 Day09 的本质是：

```python
图开始根据状态做路由决策
```

---

### Day10

Day10 很重要，因为它本质上是在做这件事：

```python
把 Day06 的手写主循环，迁到 LangGraph
```

也就是说，Day10 不是一个完全新的东西，
而是图版的“输入 -> 工具 -> 最终回复”。

所以这一段你可以直接记成：

```python
Day08-Day10 = 手写流程 -> 图流程
```

---

## 第二层：记忆系统（Day11-Day12）

这一段在回答：

```python
系统能不能记住同一个线程之前发生过什么？
```

### Day11

Day11 用的是内存型 checkpointer。

它的核心是：

- 同一个 `thread_id`
- 多次调用
- state 会接着上次继续

所以 Day11 的本质是：

```python
图有了会话级状态记忆
```

但这时记忆只是临时的，程序一停就没了。

---

### Day12

Day12 把记忆从“只在内存里”升级到“可以落盘”。

也就是说：

- 程序重启
- 图重新创建
- 只要 `thread_id` 一样
- 还是能恢复之前状态

所以 Day12 的本质是：

```python
图状态从临时记忆升级成持久化记忆
```

这一段可以压缩成一句话：

```python
Day11-Day12 = Agent 开始真正“记得住”
```

---

## 第三层：可靠性系统（Day13-Day14）

这一段不是在加花活，而是在补“真实系统必须有的兜底”。

### Day13

Day13 解决的是：

```python
工具失败了怎么办？
```

它做了三件事：

- 接住异常
- 失败重试
- 最后 fallback

所以 Day13 的本质是：

```python
工具失败 != 系统直接崩
```

---

### Day14

Day14 本质上不是新能力，而是第二阶段收口。

它在帮你确认：

- 图流程能跑
- 记忆机制没问题
- 错误处理没回归

所以 Day14 的本质是：

```python
图系统阶段的整合与回看
```

---

## 第四层：LLM 系统（Day15-Day18）

这一段开始更接近你心里想的“现代 Agent”。

它不再只是：

```python
路由 + 工具
```

而是开始进入：

```python
LLM 怎么接进来
LLM 怎么和工具协作
系统怎么处理多步任务
```

---

### Day15

Day15 先做了一件非常工程化的事：

```python
把模型调用抽成 adapter
```

这说明系统不想把模型细节写得到处都是，
而是先做一层统一封装。

所以 Day15 的本质是：

```python
模型调用标准化
```

---

### Day16

Day16 非常关键。

这一天开始出现真正像 Agent 的一层：

```python
LLM 先决定要不要调工具
-> 系统执行工具
-> 再让 LLM 生成最终答案
```

所以 Day16 的本质是：

```python
LLM 和工具真正形成闭环
```

这也是为什么你后面再去看“高级 Agent”的说法，会更容易懂。

---

### Day17

Day17 加的是 HITL，也就是人在回路。

它解决的是：

```python
某些动作不能让系统直接执行，而是要先等人批准
```

所以 Day17 的本质是：

```python
系统知道什么时候该停下来等人
```

这一步很像真实系统，而不只是 demo。

---

### Day18

Day18 不再是“一个工具干一件事”，
而是开始做多步流水线。

比如：

```python
retrieve -> calculate -> summarize
```

所以 Day18 的本质是：

```python
多个步骤的输出输入开始稳定衔接
```

这一段可以压缩成一句话：

```python
Day15-Day18 = Agent 从单步能力走向多步协作
```

---

## 第五层：工程化系统（Day19-Day21）

这一段是很多教程容易跳过，但真实工作里非常重要的一层。

它回答的问题不是：

```python
系统会不会回答
```

而是：

```python
系统能不能验证、能不能配置、能不能交付
```

---

### Day19

Day19 是评测。

它解决的是：

```python
你怎么知道这个 Agent 到底做得好不好？
```

所以 Day19 的本质是：

```python
从“感觉它还行”变成“我有方法验证它”
```

---

### Day20

Day20 做的是：

- 配置
- 环境变量
- 日志

这些东西看起来没有模型那么“炫”，
但在真实项目里非常重要。

所以 Day20 的本质是：

```python
让项目具备工程可维护性
```

---

### Day21

Day21 不是再加新知识点，
而是把前面所有东西收成一个最小毕业项目。

它做的是：

- 有统一入口
- 能跑 demo
- 能做 batch eval
- 有说明文档

所以 Day21 的本质是：

```python
把 21 天收成一个可演示、可解释、可测试的 Agent 项目
```

## 你现在应该怎么理解 Day08-Day21

如果把这 14 天压成最简总结，就是这 5 句：

1. Day08-Day10：流程图化

2. Day11-Day12：状态可记忆  
   Agent 开始按 `thread_id` 记住上下文
